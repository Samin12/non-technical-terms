Process

TLDRA program that is actually running.

Your Applications folder is a graveyard. Notes, Music, your browser — every app in there is, right now, just a file. Instructions written down and sitting on disk, doing nothing, costing nothing.

Then you double-click. The computer reads those instructions into memory, hands them CPU time, stamps the whole thing with an ID number — and now it's alive. Drawing windows. Responding to clicks. Draining your battery.

A process is a program that is actually running — the live copy in memory, not the file on disk.

The analogy that makes it stick: the file is a recipe. The process is a cake actually baking in the oven. The recipe describes a cake; the oven is where one actually exists. And one recipe can have several cakes baking at the same time.

Same recipe, two cakes

That's the trick you just pulled in the spawner. Click "Run script.js" twice and you get two rows: same name, same file — different PIDs. A PID is a process ID, the number the operating system hands every process at birth so it can tell them apart. Your real computer is running a few hundred of these right now — open Activity Monitor and you can watch them all, CPU wiggling exactly like the widget.

Because each process gets its own memory and its own slice of CPU, the two copies are complete strangers. Kill one and the other keeps running without a flicker. Real systems lean on this constantly: a busy app might run five identical copies of the same code on a server to handle traffic. Five cakes, one recipe.

Why force-quit works

When an app freezes and you reach for Force Quit, you now know exactly what you're doing: killing the process. You're pulling a half-baked cake out of the oven and binning it. Brutal — but the recipe is untouched. The program file sits on disk exactly as it was, ready to spawn a fresh process the moment you relaunch.

That's why force-quitting almost never damages an app itself. The only thing at risk is whatever the process was holding in memory and hadn't saved yet — its state. The cake is lost; the cookbook is fine. Modern apps autosave constantly for exactly this reason: they assume their process can die at any second.

Where this shows up when you build

The moment you start building with Claude, processes stop being trivia:

  • Run npm run dev and your terminal is hosting a process. Close the terminal, the process dies, your local site goes down with it.
  • Ask Claude to run a script and a process spawns, works, and exits. The runtime is the oven; the process is the bake.
  • "Port 3000 is already in use" means an old process is still alive and squatting on that port. You don't reboot — you find it and kill it.
  • Some processes are born to never exit: no window, running forever in the background. Those get their own name — daemons — and their own lesson, next.
Pro tip: when anything acts haunted — port taken, file locked, changes not showing up — ask one question first: "is there an old process still running?" Nine times out of ten the ghost is a cake you forgot in the oven. Kill the process and the haunting stops.

Once you see the split, your computer gets a lot less mysterious. The Applications folder is a cookbook. Activity Monitor is the kitchen, live, mid-service. And you now read both fluently.

Programs are promises. Processes are the work actually happening.

✓ Check yourself

1. Notes is frozen and you hit Force Quit. What did you actually destroy?

Force Quit kills the live process — the half-baked cake. The recipe (the app file) stays on disk, ready to launch a fresh process. At most you lose unsaved work.

2. You run the same script in two terminal windows at once. What does the operating system see?

One recipe, two cakes. Each launch spawns its own process with its own PID, memory, and CPU time — killing one leaves the other running, exactly like in the spawner.

3. Your dev server says "port 3000 already in use." Best first move?

A leftover process is still alive and holding the port. Kill that one process and the port frees instantly — no reboot, no re-clone, no drama.
