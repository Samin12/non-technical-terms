Env File

TLDRWhere your secret keys live. Out of the code.

You would never tape your house key to your front door. But that's exactly what hard-coding an API key is — and in the widget below you'll watch a scanner bot find it in 0.8 seconds and start spending your money.

The fix isn't to hide the key more cleverly inside the code. It's to take it out of the code entirely: key in your pocket, and a note on the door that just says "use the key from my pocket."

A .env file is a hidden file on your machine where your secrets live — the code refers to them by name and never contains them.

Try it — the leak simulator

Here's a real beginner moment: a live API key pasted straight into the code. Push it to GitHub and see what the scanner bots do. Then fix it the right way and push again.

// app.js — your code (this goes to GitHub)
const apiKey = "sk-live-4242-8819-7733";
chargeCustomer(apiKey, 49);
# .env — stays on your machine
API_KEY=sk-live-4242-8819-7733
.gitignore ✓ — never uploaded
💸 stolen-key charges: $0
The key is sitting right there in the code. Push it and find out…

0.8 seconds to get robbed is barely an exaggeration — bots scan every fresh public commit, around the clock.

What goes in it

Anything that proves you're you: API keys, passwords, database addresses with credentials inside. In the widget, the dangerous line apiKey = "sk-live-4242…" became apiKey = process.env.API_KEY — which reads as "fetch the value named API_KEY from the environment." The code now names the key; it doesn't hold the key. The actual secret sits in .env, one tidy line: API_KEY=sk-live-4242….

That tiny separation is the entire trick. Code can now travel anywhere — be shared, published, pasted into a chat with Claude — and your accounts stay locked.

Why GitHub makes this urgent

Code's natural destiny is GitHub, and a huge slice of GitHub is public. Bots crawl every fresh public commit, every minute of every day, pattern-matching for things shaped like keys: sk-live-…, AKIA…, long random strings next to the word "key." A leaked key means someone else making API calls billed to your card — the spinning charges counter was theater, but the bills are real.

The badge you saw on the .env card is the second half of the defense: .gitignore is a list of files Git must never upload, and .env goes in it on day one. That's why your second push came back clean — the code traveled, the secret stayed home.

What happens when you deploy

One catch: .env living only on your laptop means the server running your live app doesn't have it. That's by design. Every hosting platform — Vercel, Railway, all of them — has an "Environment Variables" panel in its settings. You paste the same names and values there, and your code keeps working without a single edit. Same note on the door, new pocket. More on that in the Deploy lesson — and when you start handing keys to AI agents, OAuth & Agent Identity takes this idea further: credentials live outside the code, always.

If a key touches a public repo for even a minute, treat it as burned. Deleting the file doesn't help — Git keeps history, and the bots already copied it. The recovery is always the same: revoke the key, issue a fresh one, then fix the leak. Never "it was only up briefly, probably fine."

What this unlocks for you

  • Publish or share any project without sharing your accounts
  • Run test keys locally and live keys in production — same code, different .env
  • Rotate a leaked key in one place, zero code edits
  • Let Claude read, write, and push your code freely — the secrets were never in it

This is the single cheapest disaster-prevention habit in building software. One hidden file, one .gitignore line, and the most common beginner catastrophe — the one the API lesson warned you about — simply can't happen to you.

Code is meant to be shared. Keys are not. The .env file is the wall between the two.

✓ Check yourself

1. Claude just wrote you a working app — with your Stripe key pasted right in the code. Before pushing to GitHub, you should…

The order matters: secret out of the code and .env ignored BEFORE the first push. Bots read code, not variable names — and once it's pushed, it's in history.

2. You accidentally pushed a live key to a public repo five minutes ago, then quickly deleted the file. Where do you stand?

Scanners work in seconds and Git history keeps the old version anyway. Assume it's copied: revoke, reissue, then fix the leak.

3. Your app works on your laptop but crashes on your hosting platform with "API_KEY is undefined." What happened?

That's the system working: .env stays home, so the server needs its own copy. Every platform has an Environment Variables panel for exactly this — committing .env would defeat the whole point.
