KISS

TLDRKeep it simple. Skip the clever version.

Picture two doors. One has a handle. The other has a sliding puzzle where the handle should be. Both doors open. Only one of them is a good door.

You'll test both in the widget below. Same feature — free shipping over $50 — written two ways. The simple version reads like a sentence. The clever version reads like a ransom note. And here is the trap: both passed the test.

KISS — "keep it simple, stupid" — is the rule that the simple version that works beats the clever version that works. Every time.

Try it — simple vs. clever

One feature, written two ways: free shipping on orders over $50. Test both. Then live with both.

🚪 SIMPLEreadability 
if (orderTotal > 50) {
  shipping = "FREE";
} else {
  shipping = "$5.99";
}
untested
🧩 CLEVERreadability 
shipping=[(t=orderTotal|0)>0x32?0:1,
  +!!(t>50)&1][0]?"$5.99":"FREE"
untested
⏱ 0s of dev time
Two versions. One test. Press ▶ Test both…

Both cards pass the test on day one. The gap only opens when the rule changes — and the rule always changes.

Working today is the easy part

Software has two jobs. The first is to work right now. Both of your cards did that — two green checkmarks, no difference between them. The second job is invisible: to be easy to change later. Because "later" always comes. Markets shift, rules change, marketing decides $50 should be $75.

That is where you felt the gap. On the simple card, you clicked a number and typed a new one. A few seconds, zero drama. On the clever card, you broke production twice — which 50 was the real one? what does that ? even do? — before giving up and rewriting it as, surprise, the boring version. Now multiply that by a real project: hundreds of files, dozens of rule changes a year. Clever code is not a style choice. It is a tax charged on every future visit.

And code gets visited a lot. Developers read code far more often than they write it — roughly ten times more, by most estimates. Every cryptic line gets re-deciphered by everyone who ever touches it, including the person who wrote it, six months later, remembering nothing.

Why this matters more in the AI era, not less

Claude can write genuinely clever code. Ask vaguely, and you might get the compressed, "elegant," show-off version. If you are vibe coding, that is a real problem — because you are the one left holding the puzzle door. The clever version locks you out of your own product. You cannot sanity-check it, cannot tweak it, cannot even ask a good question about it.

The simple version keeps you in the loop. You can read it. You can spot the 50 and change it yourself, exactly like you did. For a non-technical builder, readability is not a nice-to-have. It is the difference between owning your project and renting it.

The power move: ask for boring

Here is the part to actually remember. The principles in this level are not coding techniques — they are things you say. When Claude proposes something fancy, you say:

  • "Give me the boring version."
  • "Make it readable, not impressive."
  • "Would a beginner understand every line? If not, rewrite it."

"Boring" is a compliment in engineering. Boring means predictable, fixable, and calm at 2 a.m. when something breaks.

If Claude writes something you can't follow, that is not your failure. Say: "Explain this line by line, then simplify it until each explanation is one short sentence." Any line that needs a paragraph to explain is a puzzle lock — ask for the handle instead.

What this unlocks for you

  • You can read your own project — and catch problems before customers do
  • Rule changes become five-second edits, like the one you just made
  • New features stack on cleanly instead of tangling — clever code is technical debt with good PR
  • When you inherit a puzzle door, refactoring swaps it for a handle without changing what the door does

One warning before you over-apply this: simple does not mean short. The clever one-liner was shorter — and worse. Simple means obvious: the version a stranger reads correctly on the first pass. Sometimes that takes a few more lines. Spend them gladly.

Clever code impresses people for a day. Simple code keeps working after everyone forgets how it works.

✓ Check yourself

1. Claude offers two versions of your discount feature: a dense one-liner and a five-line plain version. Both pass the tests. Which do you take?

Both work today; only one is fixable on a random Tuesday. Shorter is not the same as simpler — readable wins.

2. The shipping rule changes from $50 to $75, like in the widget. In which codebase is that a five-second fix?

You watched it happen: simple was click-and-type, clever was two broken attempts and a rewrite. Cryptic code hides even its own numbers.

3. Claude proposes a "really elegant" solution you can't quite follow. What's the KISS power move?

Asking for boring is directing your AI, not doubting it. Boring means predictable and fixable — exactly what you want to own.
