Grandma's cookie recipe is perfect. You suspect peanut butter would make it better. You do not scribble on her recipe card. You photocopy it, experiment on the copy, and the original stays safe in the drawer no matter how the peanut butter batch turns out.

Software has the same tension. The project that works is precious. Experiments are how it gets better. Branches resolve it: experiment all you want, on a copy.

A branch is a parallel copy of your project where you can change anything without touching the original.

main is sacred

The top timeline in the widget — literally named main in Git — is the version of your project that works. It's what's live, what customers see, what you'd demo. While your experiment ran wild on its own track below, main kept calmly collecting its boring, reliable fixes. Two timelines, zero interference. Every dot on either track is a commit — same snapshots, just filed on different rails.

Two endings, both safe

You saw both ways an experiment can end. If it proves itself, you merge: the branch's commits flow into main with a little merge marker, and the wild idea becomes official. If it flops, you delete: the branch vanishes, and main never even knew it existed. That's the entire trick — the cost of a failed experiment is zero. The status line said it the whole time: main is safe either way.

On teams, the merge isn't a solo click. You open a pull request — "here's my branch, please review it" — and the change lands only after someone approves. Branch, then PR, then merge: that's the daily rhythm of every software team on GitHub.

This is also how you stay safe with AI. Asking Claude for a risky refactor? Have it work on a branch. The AI can be genuinely bold — rewrite whole files, try strange ideas — precisely because main can't get hurt. Branches are cheap and disposable: make one per idea, name it after the idea (try-new-pricing, fix-login), and toss the ones that don't work out.

What this unlocks for you

  • Try terrifying ideas calmly — the redesign, the rewrite, the weird experiment
  • Many things in flight at once — you on one branch, a teammate or an AI agent on another, no collisions
  • Reviewable work — a finished branch becomes a pull request someone can actually read
  • Ship anytime — main always works, so you can release even mid-experiment

The original recipe stays in the drawer. You scribble on copies — and only the improvements that earn it get written into the family cookbook.

✓ Check yourself

1. You want Claude to attempt a risky redesign while your real site keeps working. Best setup?

A branch is a full parallel copy. The AI can be bold there because the working version of your project never feels a thing.

2. The experiment flopped. What does deleting the branch cost you?

The experiment only ever existed on its own timeline. Delete it and main is exactly as it was — that zero-cost failure is the whole point of branching.

3. The experiment turned out great. How does it become part of the real product?

Merging flows the branch's commits into main, history and all. With teammates, a pull request adds a review step before the merge happens.
