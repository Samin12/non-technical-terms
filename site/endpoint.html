Endpoint

TLDROne specific address on an API.

You just clicked around a building. Hold that picture.

In the API lesson, an API was a drive-thru window — the one place your AI orders things from another app. Zoom in and the truth is slightly bigger: an API isn't one window. It's a whole building full of doors, and every single door does exactly one thing.

An endpoint is one specific address on an API — one door, one job.

The building you just explored was api.bakery.com. Knock on /orders and you get the list of orders. Knock on /orders/42 and you get one specific order. Knock on /menu and you get pastries. Same building, different doors, completely different things behind each one.

How to read an endpoint address

Every endpoint is two parts glued together, exactly like the address bar assembled itself in the widget:

  • The base URL — https://api.bakery.com. The building. It never changes.
  • The path — /orders, /customers, /menu. The door.

Now look closer at the paths, because there's a pattern hiding in them. /orders is the whole hallway — every order at once. /orders/42 is one specific room in that hallway — just order #42. Plural for the collection, an ID tacked on for one item. Nearly every API you will ever touch uses this exact shape, and that's not an accident — it's a convention, and it has a name: REST. That lesson is two doors down.

One door, one job

This is what makes endpoints powerful instead of confusing. Nobody builds a door called /do-everything. Each endpoint has one narrow purpose, which means when Stripe's docs say "send the refund to the /refunds endpoint," that's a complete, unambiguous instruction. Your AI doesn't have to guess where refunds happen. There is exactly one door for it.

It also means errors get readable. When you typed /pizza, the building didn't crash — it calmly told you 404: no such door. The internet was fine. The API was fine. You just knocked on a wall. And when you tried /orders/7, the hallway existed but room 7 didn't — same answer, smaller scale. A 404 is never mysterious once you think in doors.

One more thing before the next lesson spoils it: the same door can behave differently depending on what you do at it. Reading from /orders and creating a new order both happen at the same address — the difference is the verb you bring. That's HTTP Methods, up next.

When something fails with 404 Not Found, check the address before you check anything else. One typo in a path — /custmers — and you're knocking on a wall while the real door sits one letter away. The fix is almost never "restart everything." It's "copy the exact path from the docs."

What this unlocks for you

  • API docs stop being scary — they're door directories: address, what's behind it, done
  • You can brief Claude precisely: "call /orders/42" beats "get that order somehow"
  • Error messages turn into directions: 404 means wrong door, not broken code
  • You can predict addresses you've never seen — customers almost certainly live at /customers

Next time your AI wires up an automation and the logs scroll past, you'll see lines like GET /subscribers and they'll read like English: it went to the subscribers door.

The API is the building. The endpoint is the door. Learn to read doors and no API will ever feel like a maze again.

✓ Check yourself

1. Stripe's docs tell you to send refunds to https://api.stripe.com/v1/refunds. What is the endpoint here?

The endpoint is the one specific address: base URL plus path. Stripe's building has many doors — /charges, /customers, /refunds — and refunds happen at exactly one of them.

2. Your AI calls /orders in one step and /orders/42 in the next. What's the difference?

Plural path = the hallway, path + ID = one room in it. First call fetches every order, second call fetches just #42.

3. Your automation suddenly returns 404 Not Found on every run. Most likely culprit?

404 means "no door at this address." A dead key is a 401, and no internet means no response at all. 404 says: the building answered, and you knocked on a wall.
